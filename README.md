# AI-Powered Support Assistant

A full-stack AI customer support assistant built with React, Node.js (Express), SQLite, and the Google Gemini API. The assistant answers user queries strictly based on a provided knowledge base (`docs.json`), maintains session context, and stores all conversation history in a local SQLite database.

## ЁЯЪА Tech Stack
* **Frontend:** React.js (Vite), Axios, React-Markdown
* **Backend:** Node.js, Express.js, express-rate-limit
* **Database:** SQLite3
* **LLM Provider:** Google Gemini (`gemini-2.5-flash`) via `@google/genai`

## ЁЯЫая╕П Setup Instructions

### Prerequisites
* Node.js installed on your machine.
* A free [Google Gemini API Key](https://aistudio.google.com/app/apikey).

### 1. Backend Setup
1. Open a terminal and navigate to the backend directory:
   \`\`\`bash
   cd backend
   \`\`\`
2. Install dependencies:
   \`\`\`bash
   npm install
   \`\`\`
3. Create a `.env` file in the `backend` folder and add your API key:
   \`\`\`env
   PORT=5000
   GEMINI_API_KEY=your_gemini_api_key_here
   \`\`\`
4. Start the server:
   \`\`\`bash
   node server.js
   \`\`\`
   *(The SQLite database `chat.db` will be created automatically on the first run).*

### 2. Frontend Setup
1. Open a new terminal and navigate to the frontend directory:
   \`\`\`bash
   cd frontend
   \`\`\`
2. Install dependencies:
   \`\`\`bash
   npm install
   \`\`\`
3. Start the Vite development server:
   \`\`\`bash
   npm run dev
   \`\`\`
4. Open your browser and navigate to `http://localhost:5173`.

## ЁЯФМ API Documentation

### 1. Chat Endpoint
**POST** `/api/chat`
Processes a user message, stores it, fetches the LLM response based on `docs.json`, and returns the reply. Maintains the last 5 pairs of messages for context.

* **Request Body:**
  \`\`\`json
  {
    "sessionId": "abc-123-xyz",
    "message": "What is the refund policy?"
  }
  \`\`\`
* **Response:**
  \`\`\`json
  {
    "reply": "Refunds are allowed within 7 days of purchase. Please contact support to initiate a refund.",
    "tokensUsed": 142
  }
  \`\`\`

### 2. Fetch Conversation
**GET** `/api/conversations/:sessionId`
Retrieves the chronological chat history for a specific session.

* **Response:**
  \`\`\`json
  [
    {
      "role": "user",
      "content": "Hi",
      "created_at": "2026-02-25T10:00:00Z"
    },
    {
      "role": "assistant",
      "content": "Sorry, I donтАЩt have information about that.",
      "created_at": "2026-02-25T10:00:05Z"
    }
  ]
  \`\`\`

### 3. List Sessions
**GET** `/api/sessions`
Returns a list of all chat sessions, ordered by the most recently updated.

* **Response:**
  \`\`\`json
  [
    {
      "id": "abc-123-xyz",
      "updated_at": "2026-02-25T10:00:05Z"
    }
  ]
  \`\`\`

## ЁЯЧДя╕П Database Schema (SQLite)

The application uses two tables to persist data:

### `sessions` Table
Manages unique chat instances.
* `id` (TEXT, Primary Key): Unique UUID generated by the frontend.
* `created_at` (DATETIME): Timestamp of session creation.
* `updated_at` (DATETIME): Timestamp of the last interaction in this session.

### `messages` Table
Stores all user and assistant interactions.
* `id` (INTEGER, Primary Key, Auto-increment): Unique message ID.
* `session_id` (TEXT, Foreign Key): Links to the `sessions` table.
* `role` (TEXT): Identifies the sender (`'user'` or `'assistant'`).
* `content` (TEXT): The actual text content of the message.
* `created_at` (DATETIME): Timestamp of the message.

## ЁЯза Assumptions & Design Decisions

1. **Context Window vs. Vector DB:** I assumed the `docs.json` file represents a relatively small knowledge base. Therefore, the entire JSON file is injected directly into the LLM's system prompt. For a large-scale production app, I would implement **RAG (Retrieval-Augmented Generation)** using embeddings and a Vector Database (like ChromaDB or Pinecone) to fetch only the relevant chunks of documentation.
2. **Context Retention:** The requirement stated to keep the "last 5 user+assistant message pairs". The backend queries SQLite with `ORDER BY created_at DESC LIMIT 10`, then reverses the array to inject the most recent, relevant chronological context into the Gemini API without overwhelming the token limit.
3. **Strict Fallback:** The temperature of the Gemini model is set to `0.1`, and a strict System Instruction is applied to prevent hallucination. The LLM is forced to output "Sorry, I donтАЩt have information about that." if the answer is missing from the docs.
4. **Rate Limiting:** A basic IP-based rate limit (100 requests per 15 minutes) is implemented on the `/api/chat` route to prevent abuse and API exhaustion.
